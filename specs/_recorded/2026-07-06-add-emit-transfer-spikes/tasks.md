A judgment call was required for raw-columnar: Exasol has no native integer column type (`BIGINT` is `DECIMAL(36,0)`), so the EMITS-derived `ColumnMeta` for the `id BIGINT` column reports `Numeric`, not `Int64` — `emit_raw_columnar` strictly cross-checks the wire tag against that declared type. `emit_wide_batch_rawcol` therefore uses a dedicated `wide_row_rawcol`/`wide_types` (id as `Value::Numeric`/`ExaType::Numeric{scale:0}`) instead of the shared `wide_row`, which stays `Value::Int64` for the other (Arrow-based) paths where this check doesn't apply. A new one-off binary `benches/emit-bench/src/bin/spike_bench.rs` (not the generic shape/mode matrix) drives the arrow-ffi/raw-columnar spikes: env `SPIKE_FN` names the UDF, fixed wide shape, N=1,000,000, RUNS=1. spike-string-fast reused the existing generic `emit-bench` binary with RUNS/NS/python3/ingest temporarily trimmed to the baseline's reduced config, then reverted byte-for-byte after capture (`git diff --stat` on `main.rs` matches the pre-run 383-line diff exactly).

  baseline (no spike)  wide  row    1000000     1.525    655801    69.5   —
  baseline (no spike)  wide  batch  1000000     1.523    656486    69.6   —

  Headline finding: the hand-rolled string-block formatting fast-path (spike-string-fast) is a clear win with zero FFI/ABI surface; both FFI-based transports (arrow-ffi, raw-columnar) measured SLOWER than the existing Arrow-IPC baseline on this shape/size, likely because the per-`emit_batch_arrow_ffi`/`emit_raw_columnar_rows` call overhead (struct export/import, or the row-major encode loop) outweighs the avoided IPC serialize/deserialize round trip at this batch size (CHUNK=100,000 rows/call).
