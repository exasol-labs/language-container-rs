# Feature: version

Documents the workspace version lifecycle and the tag-triggered crates.io release flow for language-container-rs.

## Background

The workspace version is declared once in `[workspace.package].version` in the root `Cargo.toml`; all published crates inherit it, and the `exasol-udf-sdk` pin in `[workspace.dependencies]` MUST track it. This delta adds the 0.15.1 PATCH release that ships the dependency-and-dead-code cleanup with no public API change.

### Scenario: Workspace version is bumped to 0.15.1 for the dependency-and-dead-code cleanup release

* *GIVEN* the current workspace version is `0.15.0` in `[workspace.package].version` in the root `Cargo.toml` and the pinned `exasol-udf-sdk` workspace dependency is `version = "0.15.0"`
* *WHEN* the version is bumped for the `drop-unused-deps-and-dead-code` release, which removes unused dependencies, dead test fixtures, and internal dead code but introduces NO change to any published public API (`exasol-udf-sdk`, `exasol-udf-macros`, `cargo-exasol-udf`)
* *THEN* `[workspace.package].version` MUST be changed to `"0.15.1"` in the root `Cargo.toml`, and the `exasol-udf-sdk` entry in `[workspace.dependencies]` MUST be updated so its `version` field reads `"0.15.1"` to keep the pinned SDK version in sync
* *AND* `Cargo.lock` MUST be regenerated by running `cargo build` so the lock file reflects the new version; the lock-file change MUST be committed in the same PR as the version bump
* *AND* the `[workspace.dependencies].exarrow-rs` entry MUST be left unchanged (the patch release introduces no dependency-version change for `exarrow-rs`)
* *AND* the bump MUST be PATCH (`0.15.0` → `0.15.1`), not MINOR, because no public API surface changes — the cleanup is internal and manifest-level only
