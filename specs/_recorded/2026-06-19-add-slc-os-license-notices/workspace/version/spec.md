# Feature: version

Documents the workspace version lifecycle and the tag-triggered crates.io release flow for language-container-rs.

## Background

The workspace version is declared once in `[workspace.package].version` in the root `Cargo.toml`. All published crates (`exasol-udf-sdk`, `exasol-udf-macros`, `cargo-exasol-udf`) inherit this version. A release is driven by pushing a `vX.Y.Z` tag to the `main` branch; CI validates the tag matches `Cargo.toml` and publishes crates to crates.io in dependency order. v0.13.0 (execute_batch + timestamp fixes) is being tagged and released independently; the OS-license-notices work ships as the next patch, 0.13.1, and contains no API change.

### Scenario: Workspace version is bumped to 0.13.1 for the OS-license-notices release

* *GIVEN* the current workspace version is `0.13.0` in `[workspace.package].version` in the root `Cargo.toml` and v0.13.0 (execute_batch + timestamp fixes) is being tagged and released independently
* *WHEN* the version is bumped for the `add-slc-os-license-notices` release, which adds the OS-layer attribution bundle and contains no API change
* *THEN* `[workspace.package].version` MUST be changed to `"0.13.1"` in the root `Cargo.toml`
* *AND* the `exasol-udf-sdk` entry in `[workspace.dependencies]` MUST be updated so its `version` field reads `"0.13.1"` to keep the pinned SDK version in sync with the workspace version
* *AND* `Cargo.lock` MUST be regenerated by running `cargo check` or `cargo build` so the lock file reflects the new version; the lock-file change MUST be committed in the same PR as the version bump
* *AND* the `[workspace.dependencies].exarrow-rs` entry MUST be left unchanged (the patch release introduces no dependency change)
