# Feature: version

Governs the single workspace version declared in the root `Cargo.toml` and the in-sync `exasol-udf-sdk` dependency pin, plus the tag-triggered crates.io publish flow.

## Background

The workspace version is declared once in `[workspace.package]` and inherited by all published crates. This delta bumps it for the `add-multi-udf-entry-points` release. Because the change removes the bare `__exa_udf_entry` symbol and stops loading old `.so` files, it is breaking for UDF authors. The project is pre-1.0, where a MINOR bump (not PATCH) is the conventional signal for a breaking change, so `0.13.1` moves to `0.14.0`.

### Scenario: Workspace version is bumped to 0.14.0 for the multi-UDF-entry-points release

* *GIVEN* the current workspace version is `0.13.1` in `[workspace.package]` in the root `Cargo.toml` and the pinned `exasol-udf-sdk` workspace dependency is `0.13.1`
* *WHEN* the version is bumped for the `add-multi-udf-entry-points` release, which is a breaking change for UDF authors (the bare `__exa_udf_entry` symbol is no longer generated and old `.so` files no longer load)
* *THEN* `version` MUST be changed to `0.14.0` in the root `[workspace.package]`
* *AND* the `exasol-udf-sdk` entry in `[workspace.dependencies]` MUST be updated so its `version` field reads `0.14.0` to keep the pinned SDK version in sync with the workspace version
* *AND* `Cargo.lock` MUST be regenerated by running `cargo build` or `cargo update -p exasol-udf-sdk` so the lock file reflects the new version; the lock-file change MUST be committed in the same PR as the version bump
* *AND* the bump MUST be MINOR (`0.14.0`), not PATCH, to signal the breaking change to UDF authors per the pre-1.0 convention
