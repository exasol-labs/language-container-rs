# Feature: version

Documents the workspace version lifecycle and the tag-triggered crates.io release flow for language-container-rs.

## Background

The workspace version is declared once in `[workspace.package].version` in the root `Cargo.toml`. All published crates (`exasol-udf-sdk`, `exasol-udf-macros`, `cargo-exasol-udf`) inherit this version. A release is driven by pushing a `vX.Y.Z` tag to the `main` branch; CI validates the tag matches `Cargo.toml` and publishes crates to crates.io in dependency order (`exasol-udf-sdk` → `exasol-udf-macros` → `cargo-exasol-udf`). The publish step is idempotent — a version already on the index is skipped, so re-pushing a tag does not fail. The crates.io publish is the only irreversible step; all other steps (PR merge, tag push) are reversible by revert PR or tag deletion.

### Scenario: Workspace version is bumped to 0.15.0 for the dead-API removal release

* *GIVEN* the current workspace version is `0.14.0` in `[workspace.package].version` in the root `Cargo.toml` and the pinned `exasol-udf-sdk` workspace dependency is `version = "0.14.0"`
* *WHEN* the version is bumped for the `remove-sdk-dead-public-api` release, which is a breaking change for UDF authors (`column_count`, `column_name`, `column_type`, `column_index`, and `reset` are removed from `UdfContext`; `TryFrom<f64> for Decimal` is removed)
* *THEN* `[workspace.package].version` MUST be changed to `"0.15.0"` in the root `Cargo.toml`
* *AND* the `exasol-udf-sdk` entry in `[workspace.dependencies]` MUST be updated so its `version` field reads `"0.15.0"` to keep the pinned SDK version in sync with the workspace version
* *AND* `Cargo.lock` MUST be regenerated by running `cargo build` so the lock file reflects the new version; the lock-file change MUST be committed in the same PR as the version bump
* *AND* the bump MUST be MINOR (`0.14.0` → `0.15.0`), not PATCH, to signal the breaking public-API removal to UDF authors per the pre-1.0 SemVer convention
