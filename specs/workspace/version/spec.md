# Feature: version

Documents the workspace version lifecycle and the tag-triggered crates.io release flow for language-container-rs.

## Background

The workspace version is declared once in `[workspace.package].version` in the root `Cargo.toml`. All published crates (`exasol-udf-sdk`, `exasol-udf-macros`, `cargo-exasol-udf`) inherit this version. A release is driven by pushing a `vX.Y.Z` tag to the `main` branch; CI validates the tag matches `Cargo.toml` and publishes crates to crates.io in dependency order (`exasol-udf-sdk` â†’ `exasol-udf-macros` â†’ `cargo-exasol-udf`). The publish step is idempotent â€” a version already on the index is skipped, so re-pushing a tag does not fail. The crates.io publish is the only irreversible step; all other steps (PR merge, tag push) are reversible by revert PR or tag deletion.

## Scenarios

### Scenario: Workspace version is bumped to 0.13.0 for the batch-execute release

* *GIVEN* the current workspace version is `0.12.1` in `[workspace.package].version` and the pinned `exasol-udf-sdk` workspace dependency is `{ path = "crates/exasol-udf-sdk", version = "0.12.1" }`
* *WHEN* the version is bumped for the `add-connect-back-batch-execute` release
* *THEN* `[workspace.package].version` MUST be changed to `"0.13.0"` in the root `Cargo.toml`
* *AND* the `exasol-udf-sdk` entry in `[workspace.dependencies]` MUST be updated so its `version` field reads `"0.13.0"` to keep the pinned SDK version in sync with the workspace version
* *AND* `Cargo.lock` MUST be regenerated by running `cargo check` or `cargo build` so the lock file reflects the new version; the lock-file change MUST be committed in the same PR as the version bump
* *AND* `[workspace.dependencies].exarrow-rs` MUST be updated from `{ version = "0.12.7", features = ["websocket"] }` to `{ version = "^0.12.8", features = ["websocket"] }` in the same commit

### Scenario: Tag-triggered CI publishes the public crates to crates.io

* *GIVEN* a PR that bumps the workspace version to `0.13.0` has been merged to `main`
* *AND* a `v0.13.0` tag is pushed to the remote origin with `ghbrk git push origin v0.13.0`
* *WHEN* the CI `release` job runs (triggered by the `v0.13.0` tag)
* *THEN* CI MUST verify that the tag version `0.13.0` matches `[workspace.package].version` in `Cargo.toml`; if they differ the job MUST fail before publishing
* *AND* CI MUST publish `exasol-udf-sdk`, then `exasol-udf-macros`, then `cargo-exasol-udf` to crates.io in that order (dependency order)
* *AND* if any crate version is already present on the crates.io index the publish step for that crate MUST be skipped rather than failing, making re-runs idempotent
* *AND* the crates.io publish MUST be treated as the only irreversible step â€” the tag can be deleted and a corrective tag pushed if the version has not yet reached crates.io, but once `exasol-udf-sdk 0.13.0` is indexed it cannot be yanked without breaking dependents; therefore the PR MUST be reviewed and CI MUST pass before the tag is pushed

### Scenario: exarrow-rs dependency is pinned to 0.12.8 for batch-execute support

* *GIVEN* the workspace `Cargo.toml` has `exarrow-rs = { version = "0.12.7", features = ["websocket"] }`
* *WHEN* the workspace dependency is updated for the `add-connect-back-batch-execute` plan
* *THEN* the `exarrow-rs` entry MUST read `version = "^0.12.8"` (minimum 0.12.8, compatible patch range) so that `Connection::prepare`, `Connection::execute_batch_update`, and `Connection::close_prepared` from exarrow-rs 0.12.8 are guaranteed to be present at compile time
* *AND* the `features` list MUST remain `["websocket"]` unchanged
* *AND* `cargo check` MUST resolve `exarrow-rs` to a version `>= 0.12.8` in `Cargo.lock` after the update

### Scenario: Workspace version is bumped to 0.13.1 for the OS-license-notices release

* *GIVEN* the current workspace version is `0.13.0` in `[workspace.package].version` in the root `Cargo.toml` and v0.13.0 (execute_batch + timestamp fixes) is being tagged and released independently
* *WHEN* the version is bumped for the `add-slc-os-license-notices` release, which adds the OS-layer attribution bundle and contains no API change
* *THEN* `[workspace.package].version` MUST be changed to `"0.13.1"` in the root `Cargo.toml`
* *AND* the `exasol-udf-sdk` entry in `[workspace.dependencies]` MUST be updated so its `version` field reads `"0.13.1"` to keep the pinned SDK version in sync with the workspace version
* *AND* `Cargo.lock` MUST be regenerated by running `cargo check` or `cargo build` so the lock file reflects the new version; the lock-file change MUST be committed in the same PR as the version bump
* *AND* the `[workspace.dependencies].exarrow-rs` entry MUST be left unchanged (the patch release introduces no dependency change)

### Scenario: Workspace version is bumped to 0.14.0 for the multi-UDF-entry-points release

* *GIVEN* the current workspace version is `0.13.1` in `[workspace.package]` in the root `Cargo.toml` and the pinned `exasol-udf-sdk` workspace dependency is `0.13.1`
* *WHEN* the version is bumped for the `add-multi-udf-entry-points` release, which is a breaking change for UDF authors (the bare `__exa_udf_entry` symbol is no longer generated and old `.so` files no longer load)
* *THEN* `version` MUST be changed to `0.14.0` in the root `[workspace.package]`
* *AND* the `exasol-udf-sdk` entry in `[workspace.dependencies]` MUST be updated so its `version` field reads `0.14.0` to keep the pinned SDK version in sync with the workspace version
* *AND* `Cargo.lock` MUST be regenerated by running `cargo build` or `cargo update -p exasol-udf-sdk` so the lock file reflects the new version; the lock-file change MUST be committed in the same PR as the version bump
* *AND* the bump MUST be MINOR (`0.14.0`), not PATCH, to signal the breaking change to UDF authors per the pre-1.0 convention
